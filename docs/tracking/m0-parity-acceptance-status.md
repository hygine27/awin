# M0 Parity Acceptance Status

## 1. 目的

这份文档用于回答 3 个问题：

1. 当前 `awin / A视界` 是否已经达到 `M0: V1 Core Parity` 的可收尾标准
2. 这一轮实现到底改了什么
3. 还剩下哪些差异，以及这些差异是否阻塞收尾

这份文档只讨论当前已经完成的 `M0` 收口，不展开后续 `M1 / M2` 设计。

## 2. 当前验收口径

本轮采用下面这组验收口径，而不是“所有日期、所有名单 100% 一次到位”：

1. 市场层必须对齐
   - `confirmed_style`
   - `latest_status`
   - `market_regime`

2. 个股层核心 bucket 必须对齐
   - `core_anchor`
   - `new_long`
   - `short`

3. `catchup` 允许保留最后 1 个末位排序差异
   - 前提是差异已经收敛为同一类候选的末位波动
   - 不能再是框架性偏差，不能再出现大面积错位

4. 必须有脚本化验证
   - 可以对比同一交易日、同一观察时点附近的 V1 durable snapshot 与 V2 SQLite replay run

5. 必须有回归测试护栏
   - 本轮新增的排序、惩罚、重复控制、主概念优先等逻辑，都要被测试锁住

## 5. 本轮改动摘要

本轮改动的目标，不是“复制 V1 代码”，而是把 V1 合理的业务行为拉回到 V2 的结构化实现中。

### 5.1 对 `new_long` 的收口

- 保留 V1 的“主线骨干持续强势”语义，不再因为重复出现就被机械压掉
- 增加主概念优先，避免同一大主题下次级概念只靠成交额挤掉主概念候选
- 为 `catchup -> new_long` 的升级增加更严格的形态约束，防止“补涨重复出现”被误判成“顺风升级”

### 5.2 对 `catchup` 的收口

- 增加深跌反弹惩罚，避免把短线反抽误判成健康补涨
- 增加负主力净流入惩罚
- 增加 `ret_3d < 0` 惩罚，避免尚未脱离短线弱势的票占用补涨名额
- 增加“顺风强票重复占位”惩罚，避免同一只票同时过度占用 `new_long` 和 `catchup`
- 增加“新发现补涨”奖励，让新转强的补涨标的有合理上位空间

### 5.3 对验证与调试能力的收口

- 补强了 `debug_catchup.py`，支持指定 replay SQLite 库和目标股票，能直接打印关键分项
- 用脚本化 replay + parity compare 替代人工肉眼比对
- 为新增规则补足了回归测试，避免后续调整又把已对齐行为打坏

## 6. 当前回归护栏

当前本轮收口后的回归护栏是：

- 全量测试：`44 passed`
- 关键测试覆盖：
  - 高质量 repeat 的保留逻辑
  - 主线骨干连续强势保留
  - 主概念优先
  - `catchup` 的深跌反弹惩罚
  - `catchup` 的负主力净流入惩罚
  - `catchup` 的负 `ret_3d` 惩罚
  - `catchup` 的新发现奖励
  - `catchup` 对弱重复顺风票的去重占位惩罚

## 7. 当前建议的收尾口径

建议按下面这条口径宣布 `M0` 收尾：

1. `M0` 已达到“可并行验证”的收尾标准
2. V2 已承接 V1 的核心业务行为，且不是简单复制，而是以更结构化的 SQLite + replay + evaluation 方式实现
3. 当前残余问题不再是框架性偏差，而是 `catchup` 末位候选的排序波动
4. 该差异进入 `tracking`，在 `M1` 阶段通过更多交易日 replay 样本继续验证，不再阻塞 `M0`

## 8. 后续 tracking 建议

`M0` 收尾后，下一步不建议继续围绕单一样本做参数打磨，而建议转入下面 3 个动作：

1. 扩展 parity 样本
   - 选择更多交易日、更多 snapshot time 做批量 replay 对比

2. 补 `catchup` 末位排序的横向稳定性验证
   - 不只看单日单时点
   - 要看这类规则是否在不同市场环境下稳定

3. 进入 `M1`
   - 把当前已收敛的行为视为基线
   - 开始把 replay / evaluation / markdown business summary 做正式收口
