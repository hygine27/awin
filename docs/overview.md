# A视界总览

## 1. 一句话定义

**A视界（awin）是一个面向基金经理与投研团队的 A 股投资洞察平台，用于统一输出市场理解、机会发现、风险识别、持仓诊断和研究证据。**

## 2. 项目定位

A视界不是现有“风格监控”脚本的简单升级，而是一个并行演进的新项目。

它要解决的不是单一问题，而是把原来分散的几类判断放进同一个系统里：

1. 市场理解
   市场环境、大风格、主攻线、主线结构
2. 机会发现
   从 5000+ 股票中找出更值得买入或继续跟踪的标的
3. 风险识别
   识别过热、掉队、拥挤和高风险股票
4. 持仓诊断
   对已有股票池或持仓池输出继续持有、观察、减仓、退出判断
5. 研究融合
   把行情、主题、财报、研报、情报和知识库证据汇总到统一判断框架

## 3. 为什么不继续叫“风格监控”

“风格监控”这个名字太窄，无法覆盖 A视界的未来边界：

1. 不只是盘中
   还要包含跨日、复盘、回放和评估
2. 不只是风格
   还要覆盖机会发现、风险识别和持仓诊断
3. 不只是行情
   还要逐步接入财报、研报、行业材料、情报和知识库证据

所以 A视界的定位不是“监控器”，而是“投资洞察平台”。

## 4. 为什么叫 A视界

`A视界` 强调两层含义：

1. `A`
   - 指 A 股
   - 也保留主动获取超额收益的语境
2. `视界`
   - 指观察窗口
   - 指洞察视角
   - 指把市场、机会、风险和持仓放在同一视野下理解

它的语义重心不是承诺收益，而是：

**给基金经理一个更完整、更结构化的 A 股观察与判断视界。**

## 5. 与现有 V1 的关系

A视界的第一原则是：

- 不改现有 V1 脚本
- 不改现有 V1 存储结构
- 第一阶段允许只读接入外部数据源和现有运行产物
- 在自己的目录、自己的 SQLite、自己的输出协议中实现 V2

```text
现有 V1 = 现有生产监控
awin/V2 = 并行新项目，逐步验证和演进
```

V2 不是对 V1 的“就地 patch”，而是：

- 继承已有业务认知
- 在第一阶段只读复用外部输入资产
- 重新组织成可扩展、可回放、可评估的新系统

这里“独立”的含义明确为：

- 代码独立
- 规则配置独立
- SQLite 存储独立
- 输入资产暂时外接，但只能通过 `.env` 注入，不能在代码里写死外部项目路径

## 6. V2 的核心目标

V2 的核心不是“再多写几个脚本”，而是建立 4 个统一能力：

1. 统一的数据入口
2. 统一的主事实表
3. 统一的状态体系
4. 统一的业务输出

它要稳定回答 7 个核心问题：

1. 当前市场环境是什么
2. 当前大风格是谁，是否正在被挑战
3. 当前主攻线是谁，谁在加速、谁在减速
4. 从全市场股票中，哪些更值得买入或继续跟踪
5. 哪些股票更危险、拥挤或开始掉队
6. 对已有股票池或持仓池，应继续持有、观察、减仓还是回避
7. 相比上一轮、上一小时、前一日，哪里发生了真正变化

## 6.1 为什么不能停留在传统量化脚本

如果 `awin` 只是：

- 拉数据
- 计算因子
- 按规则打分
- 最后用 LLM 把结果改写成自然语言

那它本质上仍然只是传统量化程序，LLM 只承担了“翻译器”的角色。

这类系统的上限很明显：

1. 只能回答预先写进规则里的问题
2. 很难处理跨源矛盾和例外情形
3. 很难解释“为什么这次虽然分数高，但不该做”
4. 很难把研究证据、情报、财报、产业链材料纳入同一判断链
5. 很难形成“假设 -> 验证 -> 反驳 -> 修正”的判断闭环

所以 `awin` 的下一阶段方向，不应该是“把规则继续堆厚”，而应该是：

**确定性引擎负责把盘面和研究世界结构化，LLM / Agent 负责做跨源综合判断、异常审查、证据组织和交互式决策支持。**

## 6.2 LLM / Agent 时代下的定位

在这个定位下，`awin` 不再只是“市场监控脚本”，而是一个混合型投资判断系统：

1. 确定性部分负责
   - 数据接入
   - 指标计算
   - 状态识别
   - 回放与评估
   - 约束与审计
2. LLM / Agent 部分负责
   - 跨源证据归纳
   - 判断链组织
   - 例外情形解释
   - 互相质疑与复核
   - 面向基金经理的交互式问答与 drill-down

一句话说：

```text
脚本负责把事实算清楚，Agent 负责把判断做完整。
```

## 6.3 LLM / Agent 真正应发挥优势的地方

如果我是决策人，我希望 LLM / Agent 不是帮我“润色结论”，而是帮我完成下面这些传统脚本最弱的环节：

1. 跨层综合
   把市场环境、大风格、主线、资金、个股、研究证据串成一条完整判断链。
2. 证据冲突识别
   例如主线很强，但个股资金转弱；或价格很强，但研究证据不足。
3. 反方审查
   自动提出“为什么这只票虽然分高，但可能不应该做”的反证。
4. 条件化解释
   不只给固定摘要，而是能回答：
   - 为什么是它排第一
   - 如果我只看持仓会怎么判断
   - 如果我偏好低回撤，会不会换结论
5. 交互式 drill-down
   决策人可以继续追问：
   - 这条主线是不是机构主导
   - 这只票和同主题别的票比，强在哪里
   - 这轮和上一轮相比到底变了什么
6. 复盘闭环
   Agent 可以在 replay 中比较“当时的判断”和“后验结果”，给出误判归因与规则改进建议。

## 6.4 未来目标不是单 Agent，而是多角色协同

后续 `awin` 更合理的方向，不是一个“大模型统一包办”，而是多个角色围绕同一证据底座协同：

1. 市场分析 Agent
   负责市场环境、大风格、主线结构解释
2. 机会分析 Agent
   负责多头候选排序、首选依据、替代选择
3. 风险审查 Agent
   负责找反证、找过热、找价强资弱、找逻辑漏洞
4. 持仓诊断 Agent
   负责站在已有组合约束下给出继续持有 / 减仓 / 观察判断
5. 回放评估 Agent
   负责把历史 replay 结果与后验表现连接起来，推动规则与参数迭代

`openclaw`、`hermes` 或其他 agent runtime 的价值，不在于“替代数据管道”，而在于让这些角色有工具、有上下文、有审计记录地协同工作。

## 7. 第一阶段边界

### 第一阶段做

1. 市场环境判断
2. 大风格和主攻线判断
3. 股票状态机
4. watchlist 与 warning
5. 分层变化提醒
6. 日内 replay

### 第一阶段不做

1. 自动交易执行
2. 固定买卖路由
3. 固定持有天数建议
4. 多资产扩展
5. 对现有 V1 的就地替换

## 8. 第一阶段验收标准

第一阶段完成时，至少要达到：

1. 能独立在 `awin` 目录中运行
2. 能把每轮 snapshot 写入自己的 SQLite
3. 能从主表聚合出环境、风格、主攻线 summary
4. 能生成股票级状态、watchlist 和 warning
5. 能和上一轮做 diff，输出业务提醒
6. 能按交易日回放状态演进

## 9. 目标用户

1. 基金经理
2. 研究员
3. 交易支持 / 盘中盯盘人员
4. 系统维护与策略研究人员
